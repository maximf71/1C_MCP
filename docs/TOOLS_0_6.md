# Инструменты версии 0.6.0

Версия 0.6.0 добавляет шесть MCP-инструментов, совместимых по назначению с распространёнными сценариями разработки в 1C:EDT. Реализация самостоятельная: сторонний исходный код в репозиторий не включён.

## Настройка

Основные инструменты EDT включаются обычными параметрами:

```toml
args = [
  "--ditrix-edt-url", "http://127.0.0.1:8765/mcp",
  "--ditrix-project", "ИмяПроектаEDT",
  "--work-dir", "D:\\MCP-Work\\project"
]
```

Для `export_object` в форматах CF/CFE добавьте доступ к фиксированной информационной базе:

```toml
"--platform", "C:\\Program Files\\1cv8\\8.3.27.1644\\bin\\1cv8.exe",
"--infobase", "D:\\1C_Bases\\MyBase"
```

Результаты экспорта сохраняются только внутри `<work-dir>\exports`. Путь назначения нельзя передать через MCP-вызов.

Для инструмента `git` явно закрепите репозиторий и исполняемый файл Git:

```toml
"--git-root", "D:\\Sources\\My1CProject",
"--git-executable", "C:\\Program Files\\Git\\cmd\\git.exe"
```

Те же параметры доступны в команде `setup`: `--git-root` и `--git-executable`. Они сохраняются в локальном профиле без паролей и токенов.

## `list_workspace_projects`

Возвращает все проекты текущего workspace EDT, их состояние, путь и тип. Инструмент только читает модель workspace и не открывает/закрывает проекты.

## `ai_context`

Одним вызовом собирает данные объекта, формы или BSL-модуля.

- `target` — FQN метаданных или путь модуля от `src`;
- `target_type` — `auto`, `object`, `form`, `module`;
- `depth` — `minimal`, `standard`, `full`.

При `full` дополнительно запрашиваются ссылки/исходящие вызовы и диагностика проекта. Отдельные ответы EDT сохраняются без потери структурированного содержимого.

## `code_search`

Поддерживаются операции:

- `textSearch` — буквальный или регулярный поиск;
- `objectReferences` — ссылки на объект метаданных;
- `methodReferences` — вызывающие метод места;
- `resolveSymbol` — определение или hover по позиции;
- `callHierarchy` — callers/callees/outgoing;
- `help` — компактная схема операций.

Имя проекта всегда подставляется сервером. Передать другой EDT-проект через аргументы невозможно.

## `write_module_source`

Режимы: `replace`, `searchReplace`, `append`, `replaceLines`, `replaceMethod`, `insertBefore`, `insertAfter`.

Защитные меры:

- `dryRun=true` по умолчанию;
- чтение актуальной версии перед изменением;
- `expectedHash` для дополнительной проверки версии;
- точное единственное совпадение для фрагмента/якоря;
- блокировка удаления более 50% непустых строк без `allowLargeDeletion=true`;
- резервная копия в `<work-dir>\module-backups` перед реальной записью;
- запись через upstream `write_module_source` с текущим `expectedHash`;
- автоматический вызов диагностики EDT после записи.

Для настоящего изменения нужно явно передать `dryRun=false`. Рекомендуемый порядок: dry-run → проверка preview и статистики → реальная запись → просмотр диагностики.

## `export_object`

- `epf`/`erf` — сборка внешнего объекта через EDT в фиксированный каталог;
- `cf` — `/DumpCfg` фиксированной информационной базы;
- `cfe` — `/DumpCfg -Extension <имя>` фиксированной базы;
- `auto` выбирает EPF при `object_name`, CFE при `extension_name`, иначе CF.

Для CF/CFE можно передать только имя файла без каталога. Существующий файл не заменяется без `overwrite=true`. Ответ содержит размер и SHA-256 артефакта.

## `git`

Операции чтения: `status`, `log`, `diff`, `branches`.

Операции изменения: `switch`, `commit`, `push`, `pull`, `merge`, `restore`. Для каждой требуется `confirm=true`.

Ограничения:

- только репозиторий из `--git-root`;
- команды запускаются через массив аргументов, без shell;
- произвольные параметры Git не принимаются;
- пути должны быть относительными и оставаться внутри репозитория;
- `pull` и `merge` допускают только fast-forward;
- `push` использует настроенный upstream, выбрать URL через MCP нельзя;
- Git hooks отключены фиксированным пустым каталогом;
- интерактивный запрос учётных данных отключён;
- `commit` требует явные `paths` либо `all=true`.

## Приёмочная проверка

После сборки и перезапуска Codex:

1. Вызовите `list_workspace_projects` и проверьте наличие нужного проекта.
2. Вызовите `ai_context` для одного объекта и одного модуля с `depth=minimal`.
3. Выполните `code_search` с `operation=help`, затем `textSearch`.
4. Выполните `write_module_source` с `dryRun=true`; убедитесь, что файл не изменён.
5. На тестовой ветке повторите запись с `dryRun=false`; проверьте backup и диагностику.
6. Экспортируйте CF/EPF в тестовом окружении и сравните SHA-256.
7. Вызовите `git status`; затем проверьте, что изменение без `confirm=true` отклоняется.

Перед записью BSL, экспортом и Git-операциями используйте копию проекта и базы. Полный live-тест требует запущенного EDT, завершённого построения проекта и доступного `http://127.0.0.1:8765/health`.
