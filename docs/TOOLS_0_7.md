# Инструменты версии 0.7.0

Версия 0.7.0 реализует следующий уровень совместимости с типовыми сценариями разработки в 1C:EDT. Все функции закреплены за проектом, указанным при запуске MCP-сервера.

## `get_object_help`

Читает `Help/<язык>.html` объекта, возвращает исходный HTML, текст без разметки и результат `get_metadata_details`. Путь строится из FQN внутри `src` фиксированного проекта; произвольный путь передать нельзя. Пример:

```json
{"object_fqn":"Catalog.Контрагенты","language":"ru"}
```

## `launch_debugger`

Один инструмент объединяет `applications`, `configurations`, `launch`, `status`, операции точек останова, ожидание остановки, переменные, вычисление выражений, пошаговое выполнение, продолжение и завершение. Для `launch`, изменения точек останова, вычисления выражений, изменения переменных, шагов, продолжения и завершения нужен `confirm=true`.

Аргументы конкретной операции передаются во вложенном объекте `arguments` без `projectName`: сервер добавляет имя фиксированного проекта сам. У запуска по умолчанию `updateBeforeLaunch=false`, чтобы скрыто не обновлять базу.

```json
{"operation":"status"}
```

```json
{"operation":"addBreakpoint","arguments":{"modulePath":"CommonModules/Общий/Module.bsl","lineNumber":15},"confirm":true}
```

## `code_review`

Запускает отдельно установленный BSL Language Server для `src` фиксированного проекта или выбранных `.bsl`-модулей. Возвращает сводку по важности и нормализованный список диагностик. Временные исходники и отчёты удаляются после вызова.

MCP не загружает анализатор автоматически. BSL Language Server 1.0.5 требует Java 21 (Java 17 из EDT для него недостаточно). Установите официальный релиз и JDK 21, затем добавьте параметры запуска:

```toml
args = [
  "--ditrix-edt-url", "http://127.0.0.1:8765/mcp",
  "--ditrix-project", "ИмяПроектаEDT",
  "--bsl-language-server", "C:\\Tools\\bsl-language-server-exec.jar",
  "--java-executable", "C:\\Program Files\\Eclipse Adoptium\\jdk-21\\bin\\java.exe"
]
```

Необязательный `--bsl-language-server-config` фиксирует файл настроек анализатора. В вызове можно задать `module_paths`, `minimum_severity` и `limit`.

## `manage_infobase`

Доступен при подключённом EDT bridge 1.1.0 и параметре `--edt-bridge`. После обновления JAR необходимо полностью перезапустить EDT. Операции:

- `list` — зарегистрированные в EDT базы и их связь с фиксированным проектом;
- `bind` — привязать существующую регистрацию или создать файловую/серверную регистрацию;
- `unbind` — отвязать; при `unregister=true` удалить регистрацию, если она не используется другим проектом;
- `help` — краткая справка.

`bind` и `unbind` без `confirm=true` возвращают только план. Мост повторно проверяет подтверждение. Логины и пароли схемой не поддерживаются.

## `edit_metadata`

Безопасный фасад над проверенными функциями EDT-MCP для форм, расширений и СКД. Поддерживает `details`, `formStructure`, `formScreenshot`, `formLayout`, `create`, `modify`, `delete`, `setDcs`, `extensionDetails`, `extensionHandler` и `extensionResync`. Для `extensionHandler` во вложенных аргументах передаётся поддерживаемый EDT-MCP `callType`; операции расширения работают, когда фиксированный проект является проектом расширения. Аргументы исходной операции передаются во вложенном `arguments`; `projectName` задаётся сервером. Изменяющие операции без `confirm=true` возвращают dry-run.

## Ограничения безопасности

- EDT и HTTP-сервисы доступны только через loopback;
- проект выбирается при запуске сервера и не переопределяется вызовом;
- справка и модули разрешены только внутри `src` проекта;
- BSL Language Server запускается без shell-команд, по точному локальному пути;
- отладчик, метаданные и привязки баз имеют явные границы подтверждения;
- бинарники BSL Language Server и EDT-MCP не входят в репозиторий.

## Приёмочная проверка

1. Выполнить `go test ./...` и `go vet ./...` в `mcp-wrapper`.
2. Выполнить `mvn -DskipTests package` в `edt-bridge` на JDK 17.
3. На запущенном EDT вызвать `get_object_help` для объекта со справкой.
4. Вызвать `launch_debugger` с `operation=status` и `operation=listBreakpoints`.
5. Запустить `code_review` для одного небольшого модуля.
6. Проверить dry-run `edit_metadata` и `manage_infobase`; изменяющие проверки проводить только на тестовом проекте и тестовой базе.
